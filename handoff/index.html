Device Overview

Device Constraints

The target display is 800 x 480 px in landscape orientation.

There are five physical reader ports below the screen, so the UI often needs to show both the on-screen workflow state and the physical cassette state at the same time.

Assume this is an Android device. Where possible, use standard Android behavior for the keyboard, date and time pickers, navigation, and opening new screens instead of redesigning those patterns.

  • If a screen can fit without scrolling, that is preferred.
  • Scrolling should be limited as much as possible because the touchscreen does not scroll well.
  • The most important actions and the port status should be readable at a glance.

Normal Test

A normal test is the standard workflow for a milk sample. The user chooses a port, enters sample and operator information, selects or confirms the test type, and starts the read.

Some tests are read only. Other tests need incubation before reading.

  • Configuration can happen before the cassette is inserted.
  • The test type can be chosen manually or prefilled from QR.
  • The main goal is to help the operator prepare the correct test on the correct port with as little friction as possible.

Confirmation Flow

If the first result needs confirmation, the device guides the operator to Test 2 and sometimes Test 3.

A modal appears only when the operator needs to make the next decision or take the next step. If the flow is already complete, no extra modal is shown.

  • Test 2 and Test 3 always require a newly inserted cassette.
  • The next cassette must be the same test type as the original test.
  • If Test 1 is negative, the flow ends without a confirmation modal.

Controls

Positive controls and animal controls are not normal sample tests. They are special runs where a positive result is expected.

They should stay clearly separated from normal tests in both the workflow and the history.

Verification

Verification is a service workflow used to check port performance and reading behavior.

The operator places a physical thermometer into the port, measures the real temperature, and compares it to the selected 40 C or 50 C target.

The result also shows values such as light intensity and line ratios. These help show how well the device can read the cassette lines.

  • All ports always share the same reader temperature setting.
  • Each port should be verified every 250 tests.
  • The user can set a lower local warning threshold, but the cloud default stays 250.
  • After verification, the counter for that port resets to zero.

Cassettes, Strips, and QR

The device supports both strip tests and cassette tests. Strips are the older format and do not have QR. Cassettes are the newer format and can have QR.

When QR is enabled and the cassette supports it, the device reads the test type from QR and does not let the user change it.

This matters because different test types can have different substances, different cassette layouts, and different numbers of lines.

During confirmation flows, the device also checks that the next cassette is the same test type as the original test.

If QR is turned off, the user must choose the test type manually.

  • A batch means a box of many cassettes of the same type.
  • The slot can show the control line and the test lines for the inserted cassette.
  • Test lines can use short substance abbreviations because some tests have fewer lines and some have more.
  • The control line is always present because it shows whether the test itself worked.

Microswitch

The microswitch is the cassette insertion sensor.

Its main job is to detect that a cassette was inserted after configuration so the read action can be enabled.

Its limitation is that it can reliably detect insertion, but not removal.

When it is on, the read buttons unlock only after cassette insertion is detected. When it is off, the device allows a fallback mode without that check, for example if the sensor is broken.

Account and Settings Access

The device can be signed in to the cloud or used in anonymous mode.

In anonymous mode, test records are not sent to the cloud.

When the device is signed in, the allowed test types come from the backend. When it is anonymous, the user can turn test types on and off locally in Settings.

  • Some settings are open to normal users.
  • Some settings require the reader password.
  • After the password is entered, protected settings stay unlocked while Settings remains open.

Other Important Concepts

  • Some tests require a specific reader temperature, usually 40 C or 50 C.
  • Quantitative tests need a loaded quant curve before they can start.
  • History separates normal tests and controls.
  • Exports are CSV or LIMS, not Excel.
  • Root full-screen screens use Close, while pushed child screens use Back.
