Overview

Device Overview

Use this page as a quick briefing for a designer who has not seen the reader before. It explains what the device does, why some flows exist, and which hardware constraints shape the interface.

Device Constraints

The target display is a small landscape touchscreen at 800 x 480 px.

There are five physical reader ports below the screen, and the UI often needs to reflect both the on-screen state and the physical cassette state at the same time.

Assume this is an Android-based device. Native Android patterns such as the keyboard, date and time pickers, screen transitions, modal presentation, and the system navigation area should be used where possible instead of being heavily redesigned.

  • Design for fast reading at arm's length.
  • If something can fit on one screen without scrolling, that is preferred.
  • The touchscreen has poor scrolling behavior, so avoid scrolling unless the screen truly needs it.

Normal Test

A normal test is the standard sample workflow. The user configures a port, enters sample and operator information, confirms or selects the test type, inserts the cassette, and starts reading.

Some tests are read only, while others need incubation first. A positive or conflicting result can trigger a confirmation flow with Test 2 and Test 3.

The confirmation flow exists only when the next operator action is needed. For example, if Test 1 is negative, the flow is complete and no confirmation modal is shown. If Test 1 is positive, or if Test 2 conflicts, the UI asks the operator to continue with the next test.

  • Configuration may happen before cassette insertion.
  • Test type can be manual or prefilled from cassette QR.
  • Confirmation tests always require a newly inserted cassette.
  • Test 2 and Test 3 must stay tied to the same test type as the original test.

Controls

Controls are not normal sample results. They are special runs, mainly positive controls and animal controls.

These are tests where a positive outcome is expected, so they should stay clearly distinct from normal sample tests.

They should remain clearly distinguishable from regular tests in both the workflow and the history.

Verification

Verification is a service or maintenance workflow used to check port performance and reader temperature behavior.

During verification, the operator uses a physical thermometer to measure the temperature inside the selected port and compares that measured value to the selected 40 C or 50 C target.

All ports always use the same reader temperature setting. The rest of the verification result is based on device outputs such as light intensity and line ratio values, which show how well the reader can read the cassette lines.

Each port should be verified every 250 tests by default. The local warning threshold can be set lower on the device if the user wants to see the warning sooner, but the cloud-level expectation remains 250.

Once verification is completed, that port's verification count resets to zero.

QR Scanning

The reader supports both strip tests and cassette tests. Strip tests are the older format and do not have QR. Cassette tests are the newer format and can include QR.

When QR scanning is enabled and a supported cassette is inserted, the reader uses the QR to identify the test type and prefill it so the operator cannot change it.

This is important because different test types can have different substances, different cassette layouts, and different numbers of lines.

During confirmation flows, the reader also checks that the next cassette is for the same test type as the original test.

When QR scanning is off, the operator must choose the test type manually.

  • A cassette belongs to a batch, meaning a box of many cassettes of the same type.
  • The slot visualization should show the control line plus the test lines for that cassette.
  • The test lines can be labeled with short substance abbreviations because some tests have fewer lines and some have more.
  • The control line is always present because it shows whether the test itself worked.

Microswitch

The microswitch is the reader's cassette insertion sensor.

Its main use is to detect that a cassette was inserted after configuration so the reader can enable the read action.

The limitation is that it can detect insertion, but it cannot reliably detect removal.

When microswitch checking is on, the read actions stay disabled until a cassette is detected. When it is off, the reader allows a fallback service mode where the read buttons stay enabled even without insertion in case the microswitch is broken.

Accounts And Settings Access

The reader can be used either signed in to the cloud or anonymous.

Anonymous mode means the device does not send test records to the cloud.

When the reader is signed in, the allowed test types for that device come from the back end. When it is anonymous, the operator can locally enable or disable test types in Settings using on and off toggles.

  • Some settings are open to normal users.
  • Some settings require a reader password.
  • Once the password is entered, protected settings stay unlocked until the user leaves Settings.

Other Important Concepts

  • Some tests require a specific reader temperature, typically 40 C or 50 C.
  • Quantitative tests need a loaded quant curve before they can start.
  • History separates Tests and Controls.
  • Exports are CSV or LIMS, not Excel.
  • Root full-screen views close, while pushed child views use back navigation.
